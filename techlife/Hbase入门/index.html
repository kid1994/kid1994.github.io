简介

上面提到，NoSQL是为了解决传统关系型数据库所存在的问题，这些问题主要包括以下几个方面：

  1. 传统关系型数据库主要适合事务性操作，但是不擅长进行大规模数据的分析处理，因为大规模的数据查询需要进行大范围的数据扫描或全表扫描。
  2. 传统关系型数据库在事务和并发增加时，所带来的等待和死锁的出现频率会急剧增长。

而传统关系型数据库自己解决上述问题的方式，通常是以牺牲部分关系型特性为代价，例如辅助索引，来换取更高的性能。或者用户通过反范式化数据模型来避免等待，以及通过降低锁的粒度来减小死锁出现的频率。

于是，针对不同具体的业务场景和需求，便出现了NoSQL数据库。

为特定的问题制定差异化的专用解决方案。

我们可以根据以下角度或维度来区分各个NoSQL数据库：

  1. 数据模型。指数据的存储方式，是键值对型、半结构化的列式存储还是文档结构存储。
  2. 存储模型，是内存存储还是持久化存储。
  3. 一致性模型，一致性的严格程度由强到弱可分为：
    1. 严格一致性：数据已经改变，即时生效。
    2. 顺序一致性：数据的状态根据对数据操作的执行顺序而变化。
    3. 最终一致性：在没有数据更新的时间内，系统通过广播来保证各个数据副本之间的一致性。
    4. 弱一致性：系统通过广播的形式来传播对数据的更新，不能保证各个外部用户观察到数据更新的先后顺序。
  4. 物理模型，系统架构是单机模式还是分布式模式。
  5. 读写性能，支持读多写少、读写相当还是读少写多。
  6. 是否支持辅助索引。
  7. 是否支持故障处理。
  8. 是否支持数据压缩。
  9. 是否支持负载均衡。
  10. 是否支持原子性的读写操作。
  11. 支持的锁模型。

而就这些维度来描述HBase而言：

HBase是一个分布式的、持久的、强一致性的存储系统，具有近似最优的写性能（能使I/O利用率达到饱和）和出色的读性能，它充分利用了磁盘空间，支持特定列族切换可选压缩算法。

在实际选择NoSQL的时候，也要从特定的需求出发，也不一定传统关系型数据库不能解决的问题，用NoSQL就能完美地解决。

安装过程这碰到的问题

问题：日志中报错IOException : cannot create directory
解决方法：目录没有写权限。修改目录权限，然后使用root账户进行操作就可以了。
